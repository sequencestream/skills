# ss-sdd 设计深度解读

> 本文以 AI Agent 工程实践者视角，对 `SKILL.md` 中的每一个关键设计进行解构与剖析，揭示其背后的权衡（trade-off）、假设与适用边界。

---

## 一、整体设计哲学：在"模型自驱"与"人类掌控"之间寻找最小公倍数

ss-sdd 不是又一个 BMAD/Spec-Kit 式的重型流程框架，也不是放任模型自由发挥的 vibe coding。它的核心立场可以一句话概括：

> **承认模型是进度的主驱动力，但在关键时刻用最小的控制门维持人的方向权。**

这个立场的深层判断有三条：

1. **重流程会被 token 经济学反噬**：在多轮、高输入的编码任务中，长 prompt、长 plan、长 RIPER 文档本身就是高昂的"常驻 token 成本"，最终被上下文压缩或腐烂（context rot）抹掉，得不偿失。
2. **轻流程会被信息熵反噬**：完全没有 spec 的对话式编码，会随对话轮次增加快速失去锚点，模型开始幻觉历史、重复已废弃方案、忘记早期约束。
3. **真正稀缺的是"对齐时刻"**：人类不需要审计每一步，只需要在 5 个时刻（Restate / Checkpoint / Approval / Validation / Reverse Sync）确认方向没歪。

于是 ss-sdd 选择了**门控（gating）而非细化（specifying）**的控制范式 —— 这是它与 BMAD、Spec-Kit、Kiro 等同类方案最本质的差异。

---

## 二、核心定位：从"助手"到"主驱动力"的范式转换

```
模型：分解、探索、试验、推进
人类：方向、边界、节奏、风险、验收
```

### 关键设计点

- **不再让人类做计划，而是让人类做评审**。这是对"人在回路（Human-in-the-loop）"的重新定义：人类的角色不是"参与每一步"，而是"在合适的边界上 yes/no"。
- **"控制不来自预先指定每一步"** —— 这是对传统软件工程"先设计后编码"思维的反叛。在 LLM 时代，预先指定的细节往往不如即时探索得到的细节准确，因为代码现状（reality）本身就是最丰富的设计文档。
- **"目标不是减少控制，而是减少低价值常驻 token"** —— 这句话点破了所有重型 AI 流程框架的痛点：它们用大量 token 维持"看起来可控"的假象，而实际控制力来自人对结果的判断能力，不来自流程长度。

### 工程含义

这意味着 ss-sdd 的真正成本中心不是"写规约的时间"，而是"决定何时该停下来对齐"的判断力。skill 的所有设计都在为这个判断力服务。

## 三、五个控制门（Five Gates）：最小可行控制集

| 门 | 时机 | 解决的失效模式 |
|---|---|---|
| **Restate** | 收到任务后 | 模型误解任务（最高频失效，廉价拦截） |
| **Checkpoint** | 执行前 | 方向悄然漂移、风险未被识别 |
| **Approval** | 执行前 | 不可逆动作未经授权 |
| **Validation** | 执行后 | 模型自我宣称"完成"但未被外部证据证伪 |
| **Reverse Sync** | 执行后 | 知识只存在于对话上下文中，下一轮即丢失 |

### 为什么是这 5 个，不是 7 个或 3 个？

这是经过失效模式分析（failure-mode analysis）后留下的**最小必要集**：

- **Restate** 拦截"输入侧"误解 —— 据社区经验，这是 LLM 编码失败案例的最大单一来源。
- **Checkpoint** 拦截"启动侧"漂移 —— 在多轮长对话中，模型自身会偏离最初目标。
- **Approval** 拦截"高风险动作" —— 与 Claude Code 系统提示中的"executing actions with care"原则共振。
- **Validation** 拦截"假完成" —— LLM 有强烈的"宣称成功"倾向（reward hacking 的轻量形态）。
- **Reverse Sync** 拦截"知识蒸发" —— 跨会话、跨人协作时的关键环节。

少一个都会留下不可控的漏洞，多一个都会回到 RIPER 那种"为流程而流程"的状态。

### 深层设计：门是离散的，不是连续的

ss-sdd 没有要求模型"持续汇报进度"，而是要求模型"在 5 个离散时点停下来"。这是典型的**事件驱动控制**而非**轮询控制** —— 既贴合人类注意力的离散特性，又避免了对话被过度填充。

## 四、规约（Spec）的双重读者设计

> **"规约的首要读者是人；模型是第二读者。"**

这是 ss-sdd 最反直觉、也最关键的一个设计判断。

### 为什么这个顺序非常重要？

大多数"AI 工程化"流派把规约视为"给模型读的精确指令"。这导致两个后果：

1. 规约越来越像 DSL，人类读起来反而吃力。
2. 规约的所有价值都绑定在"模型这次是否会读它"上，会话切换或模型升级后即失效。

ss-sdd 反过来定义：
- **规约首先是持久化的任务上下文与组织记忆** —— 它是给"未来的人"和"协作的人"看的。
- 模型只是顺便受益的"第二读者"。

### 规约对模型的四重价值（针对模型这个"次要读者"的设计）

| 价值 | 对抗的失效 |
|---|---|
| **Attention focus** | 模型注意力被无关上下文稀释 |
| **Information index** | 全文重读浪费 token，按需查找更优 |
| **Defense against context rot** | 长对话中模型遗忘早期约束 |
| **Review aid** | 缺乏 spec-vs-code 的交叉校验基线 |

这四点合起来，定义了**模型究竟如何使用规约**：**不是把它当作执行指令逐句翻译，而是当作可索引、可对齐、可回查的知识库**。

### 工程含义

规约文件应该写得**像给资深同事的交接文档**，而不是像给编译器的输入。这反过来约束了 ss-sdd 强烈推荐的写作风格：自然语言、结论先行、面向 reviewer。

## 五、九条硬约束（Hard Constraints）的体系学解读

九条硬约束并非平铺并列，而是构成了一个闭环：

```
   ┌──── Spec is Truth ────────────┐
   │                                │
   │   ┌── Restate First            │
   │   │      ↓                     │
   │   │   No Spec, No Code         │
   │   │      ↓                     │
   │   │   Core Goal as Loop Anchor │
   │   │      ↓                     │
   │   │   Checkpoint Before Execute│
   │   │      ↓                     │
   │   │   No Approval, No Execute  │
   │   │      ↓                     │
   │   │   Done by Evidence         │
   │   │      ↓                     │
   │   │   Reverse Sync ─────────┐  │
   │   │      ↓                  │  │
   │   └── Resume Ready          │  │
   │                             │  │
   └─────────────────────────────┘──┘
```

逐条剖析：

### 1. `Spec is Truth`（规约即真理）
**设计意图**：解决"上下文窗口外的状态丢失"问题。在多轮对话中，唯一不会被压缩、不会被 LLM 遗忘的就是写在磁盘上的文本。这条规则强制把"事实"外置到文件系统，把对话变成可消散的临时介质。

**反例代价**：不遵守时，会出现"我们之前讨论过的 X 呢？" → 模型只能编造或承认遗忘。

### 2. `No Spec, No Code`（无规约不编码）
**设计意图**：拒绝"边想边写"导致的方向漂移。这不是"瀑布"，而是要求**至少有一个最小锚点**存在于会话外。

**关键克制**：规约可以是 3 行，可以是 30 行，但不能是 0 行。这与 Spec-Kit 要求"完整规约"形成鲜明对比。

### 3. `No Approval, No Execute`（无批准不执行）
**设计意图**：把"不可逆动作"从模型的自主决策中剥离。这是 trust boundary 的物化。

**深层考量**：approval 不是形式主义，而是给人类一个"低成本叫停"的窗口 —— 在执行前叫停的代价远小于回滚的代价。

### 4. `Restate First`（先复述）
**设计意图**：把"理解错误"从执行后才发现，前移到第一句话就暴露。
**实证支撑**：所有研究 LLM 编码失败的工作都指向同一结论 —— **任务理解错位是 #1 错误源**。Restate 用一段廉价文本拦截掉它。

### 5. `Core Goal as Loop Anchor`（核心目标作为循环锚点）
**设计意图**：解决"循环漂移"问题 —— 模型在多步推进中，目标会被"附带优化"、"顺便重构"、"修复发现的旁路问题"等慢慢稀释。这条约束要求**每次执行前后都用核心目标做一次重对齐**。

**类比**：相当于船员在长航行中反复校对罗盘，而不是只在出港时看一次。

### 6. `Checkpoint Before Execute`（执行前打检查点）
**设计意图**：在"思考"和"动手"之间插入一个**结构化暂停**。Checkpoint 不是流程负担，而是模型把"我现在准备做什么"显式化的机会 —— 显式化即可被审查。

**结构强制**：理解 / 目标 / 下一步 / 风险 / 验证方法 —— 这五要素是被精挑细选的，覆盖了"任务质量评审"的最小信息集。

### 7. `Done by Evidence`（用证据证明完成）
**设计意图**：对抗 LLM 的"宣称成功"倾向。模型说"已完成"不算完成，测试通过、日志符合预期、人类确认才算。

**深层含义**：把验证标准外置到 spec 里（`Done Contract`），让"什么叫完成"不再由模型当场定义。

### 8. `Reverse Sync`（反向同步）
**设计意图**：闭环。**模型对真实世界做了改变，必须把这个改变的事实、偏差和验证结论写回规约**。

**容易被忽略的价值**：这是把"对话中的隐性知识"显性化的关键步骤，没有它，下次新会话就要从零开始。

### 9. `Resume Ready`（可恢复）
**设计意图**：为"长任务被打断"做准备。任何随时可能被打断的工作，都需要留一个"下次从哪里继续"的锚点。

**协作含义**：让 spec 不仅服务于当前会话的恢复，也服务于交接给另一位开发者（或另一个 AI 实例）。

## 六、默认假设（Default Assumptions）：反仪式化的克制

```
- 模型自行分解任务，不要强行展开 RIPER 五段式
- 不要为协议拆分小任务，也不要为仪式给大任务写长计划
- 只重读当前相关章节，不要重新加载整个文件
```

这三条共同构成了 ss-sdd 的**反仪式宣言**。

### 关键洞察

- **流程粒度应与任务复杂度匹配**，这是显然的。但在大多数 AI 工程框架中，反复出现"小任务也走五段式"的过度合规问题 —— 它本质上是把"流程合规"误当作"质量保证"。
- ss-sdd 把"动态调整流程粒度"显式化为 `Task Depth` 四档分级，用配置选择代替默认膨胀。

### "只重读当前章节"为什么重要

这是对**token 经济学**的直接回应。在长 spec 文档中，反复全文重读：
- 每次都消耗大量 token
- 注意力被稀释到无关章节
- 容易触发模型对早期版本内容的"幻觉激活"

正确做法是**索引式访问**：只读当前在做的那一块，其他章节作为"按需查询的备查"。

## 七、任务深度分级（Task Depth）：四档而非三档的精妙之处

```
zero      → 不写规约直接做（机械变更）
fast      → 微规约（1-3 句话）
standard  → 默认档（轻量规约）
deep      → 深度档（架构、模糊需求、跨模块）
```

### 为什么是四档？

大多数同类方案给出三档（简单/中等/复杂），ss-sdd 多了一个 `zero` 档。这个选择背后是一个**重要的实证观察**：

> 在真实编码任务中，存在大量"没有决策内容"的纯机械变更（typo、log、配置改值）。强迫它们走规约流程，是协议本身在制造无意义负担。

`zero` 档承认这种存在，并给它一个**正式的"无规约通道"**。但这个通道有一个安全阀：**"如果复杂度超出预期，立即升档"** —— 这避免了"为了快而忽视真实风险"的陷阱。

### 档位的真实使用边界

| 档位 | 真实代表的复杂度信号 |
|---|---|
| `zero` | 改一个单词、一个字面量、一行日志 |
| `fast` | 1 个文件内的局部修改，逻辑可被一句话描述 |
| `standard` | 2+ 文件，需要思考接口或调用关系 |
| `deep` | 需要做技术选型、要权衡 trade-off、根因未知 |

### 档位选择由谁决定？

skill 的措辞是：**用户可以在 prompt 中给提示（如 `/ss-sdd fast: ...`），否则模型自行选择**。这是一个**双方都可主导的决策点**，比强制由人选或强制由模型选都更灵活。

## 八、最小工作流（Minimal Workflow）：闭环六步的精度

```
1. Restate（复述理解）
2. Spec（写入最小规约，含 Done Contract）
3. Checkpoint（执行前检查点）
4. External evidence check（外部证据校验，发现偏差则重新对齐）
5. Approval & Execute（批准后执行）
6. Reverse Sync（写回变更日志、验证结论、恢复锚点）
```

### 第 4 步的隐藏价值

很多读者会一带而过第 4 步：
> "如果测试、日志、人类反馈暴露偏差，基于外部证据重新陈述'当前核心目标是否变化、还缺什么'，再决定继续或调整。"

这一步是 ss-sdd **真正不同于线性瀑布的关键** —— 它显式建模了**"循环中的中断与重对齐"**。模型不是从 1 走到 6 就结束，而是在 4 这个节点上有可能跳回 1 或 2 或 3，形成 OODA 风格的快速循环。

### `Done Contract` 的作用

在第 2 步要求用 1-3 行写下：
- 什么算完成
- 什么证据可以证明
- 什么状态算"尚未完成"

这是把验证标准前置化 —— 写在 spec 里，而不是等执行后让模型自己定义。它防止了"完成定义随结果漂移"。

## 九、暂停机制（When to Pause）：六个红灯

```
1. 关键歧义会改变实现方向
2. 需要破坏性、高风险、不可逆操作
3. 触及架构、公共接口、数据模型、迁移策略
4. 跨项目且未明确授权或边界不清
5. 现有规约明显错误、过时、与代码现实冲突
6. 尚未形成最小规约或未获得明确执行批准
```

### 设计逻辑

这六条覆盖了**"模型不应当自主决定"的所有典型场景**：
- 1, 5 是认知边界（不清楚 → 暂停）
- 2, 3 是风险边界（高代价 → 暂停）
- 4 是授权边界（越界 → 暂停）
- 6 是程序边界（流程未走完 → 暂停）

### 与 Claude Code 系统提示的对齐

Claude Code 系统提示有一段"Executing actions with care"，提到"对不可逆的行为要先确认"。ss-sdd 把这个原则**结构化、显式化**到了 skill 层级 —— 这是一种**良好的层级耦合**：底层有原则，skill 层把原则可执行化。

## 十、Spec 路径约定（Spec Path）：日期化的目录结构

```
<root-session-dir>/<YYYY>/<MM>/<DD>/YYYY-MM-DD-<NNN>-<requirement-short-name>/
```

### 设计意图

- **日期分桶**：自然形成时间序列的可追溯归档。
- **NNN 序号**：同一天内多个任务时不会撞名。
- **短名标识**：人类扫一眼就知道这是什么任务。
- **默认 `changes/`**：与开源项目常见 `docs/`、`adr/` 等约定不冲突。

### 隐含的组织含义

这种结构暗示了一种**轻量级 ADR（Architecture Decision Record）替代品**的定位 —— 每一次 ss-sdd 介入都留下了一份可被组织化检索的"决策与变更记录"。

这是 ss-sdd **真正能沉淀为组织记忆**的地方：不是依赖外部工具（Linear、Jira），而是直接把记忆物化为仓库内的文件，与代码版本一致演进。

---

## 十一、与同类方案的对比定位

| 方案 | 流程粒度 | 规约形态 | 控制方式 | ss-sdd 的差异 |
|---|---|---|---|---|
| BMAD | 重 | 多文档 | 多阶段 gating | ss-sdd 把多阶段折叠为 5 个 gate |
| Spec-Kit | 中-重 | 完整 spec | spec-first | ss-sdd 接受"3 行 spec"作为合法形态 |
| Kiro | 中 | 工件链 | 工件驱动 | ss-sdd 不强制工件链，按需写 |
| RIPER | 中 | 五段式 | 状态机 | ss-sdd 默认不展开 RIPER，复杂时才升档 |
| Vibe coding | 零 | 无 | 无 | ss-sdd 强调最小可行 spec |

**ss-sdd 占据的是"轻量但有骨架"这个生态位** —— 它认为既有方案要么过重，要么过轻，缺少一个"轻得几乎无感、但关键时刻给力"的中间解。

---

## 十二、对落地者的几点关键提醒

1. **不要把规约写成代码 DSL** —— 它的首要读者是人，包括 3 个月后的你自己。
2. **不要在 zero 档伪装成 fast 档** —— 真的机械变更就直接做，不要为了"看起来专业"而强加规约。
3. **Restate 的语言风格应当是"我自己的话"** —— 复读用户原文等于没复述。
4. **Done Contract 要写得可证伪** —— "实现 OAuth"不够，"`/auth/refresh` 在 token 过期 1 分钟内返回新 token 且日志包含 refresh_id"才够。
5. **Reverse Sync 不是"任务总结"** —— 它是把"对真实世界的变更与发现"写回规约的工程动作，重点是**偏差**和**验证结论**，不是把对话历史抄过去。
6. **遇到红灯就停** —— 暂停的代价远低于回滚的代价；不要因为"已经走到这一步了"而硬推。

---

## 结语

ss-sdd 的核心智慧不在于它发明了什么新概念，而在于**它对每一个旧概念都做了精确的"减法"**：

- 把规约减到最小可行的几行
- 把流程减到 5 个离散门
- 把仪式减到"必要时才升档"
- 把控制减到"关键时刻的人类 yes/no"

这种克制的本质是承认了一个事实：**在 LLM 时代，过度的流程化是新形态的过度工程**。真正的工程纪律不是"做了多少检查"，而是"在对的时刻做了对的检查"。

ss-sdd 把"对的时刻"识别为 5 个 gate，把"对的检查"压缩到极简的复述、检查点、批准、证据验证、反向同步 —— 这就是这份 skill 的全部设计精华。
